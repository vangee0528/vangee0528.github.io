Flutter 状态管理方案对比

Flutter 2026-02-05 · 16 分钟阅读

Provider、Riverpod、Bloc、GetX —— 四大状态管理方案的原理与适用场景分析。

为什么需要状态管理？

Flutter 是声明式 UI 框架，UI = f(state)。当状态改变时，UI 需要重新构建。良好的状态管理可以让代码更清晰、可维护性更高。

Provider

官方推荐的状态管理方案，简单易用。

// 定义
class Counter extends ChangeNotifier {
  int _count = 0;
  int get count => _count;
  void increment() {
    _count++;
    notifyListeners();
  }
}

// 使用
Consumer<Counter>(
  builder: (context, counter, child) {
    return Text('${counter.count}');
  },
)

优点：官方支持，学习曲线平缓，适合小型项目。

缺点：需要手动调用 notifyListeners，测试略麻烦。

Riverpod

Provider 的升级版，解决了 Provider 的一些问题。

// 定义
final counterProvider = StateProvider((ref) => 0);

// 使用
Widget build(BuildContext context) {
  final count = ref.watch(counterProvider);
  return Text('$count');
}

优点：无需 BuildContext，编译时安全，易于测试。

缺点：概念较多，学习曲线稍陡。

Bloc

基于 Stream 的状态管理，强调事件驱动。

class CounterBloc extends Bloc<CounterEvent, int> {
  CounterBloc() : super(0) {
    on<Increment>((event, emit) => emit(state + 1));
  }
}

// 使用
BlocBuilder<CounterBloc, int>(
  builder: (context, count) => Text('$count'),
)

优点：架构清晰，适合大型项目，易于追踪状态变化。

缺点：样板代码较多。

GetX

全功能框架，包含路由、依赖注入、状态管理。

// 定义
class Controller extends GetxController {
  var count = 0.obs;
  void increment() => count++;
}

// 使用
Obx(() => Text('${controller.count}'))

优点：极简代码，功能全面。

缺点：过于"魔法"，不利于长期维护。

方案对比

方案 复杂度 适合项目
Provider 低 小型项目
Riverpod 中 中小型项目
Bloc 高 大型项目
GetX 低 快速原型

总结

选择状态管理方案需要根据项目规模和团队习惯。建议初学者从 Provider 入手，掌握后再探索其他方案。
